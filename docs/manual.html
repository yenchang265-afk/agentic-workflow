Supervised automation for long-lived goals

Run a goal like a state machine, not a chat.

agentic-loop drives a goal through named stages automatically, pausing only at the human gates you decide matter. It is a multi-kind loop framework: each loop kind is a declarative manifest — stages, transitions, a work source — interpreted by one shared engine and fed by a common scheduler.

engineering · default-on · PLAN→BUILD→VERIFY→REVIEW pr-sitter · opt-in · watches your open PRs ships as 2 plugins · OpenCode + Claude Code
⊘Human gatesThe agent parks at review points; you approve before it moves on. It never merges on your behalf.
⑃Git isolationCode-writing stages run on a loop branch or an isolated worktree — your working tree is untouched.
✓Trusted verdictsPASS / FAIL / ERROR count only through a dedicated tool. A stage that just says "passed" is ignored.
▤Audit trailEvery run writes a summary — per-stage wall-clock, verdict history, iterations used — committed alongside the work.
New here? Read top to bottom: install it, meet the config wizard, then see how the loop actually moves and what each skill contributes. Reference tables sit at the end.
Setup

Quickstart & install

Two install routes. Use bootstrap.sh on a fresh machine (it installs system prerequisites for you); use the manual route if you already have the toolchain.

Prerequisites

ToolNeeded forNote
Node.js ≥ 20Core & both pluginsHard requirement.
git, curlIsolation, fetchingAlways.
ghGitHub PR workAuthenticate with gh auth login.
az + azure-devops extAzure DevOps PRsOptional — skip with --no-ado.
Chrome / ChromiumBrowser UI testingOptional — skip with --no-browser.

bootstrap.sh verifies/installs the system CLIs, registers the external MCP servers (Chrome DevTools, ADO), then hands off to install.sh all. It never automates auth — it prints reminders at the end.

shell
./bootstrap.sh                 # everything
./bootstrap.sh --no-ado        # skip Azure CLI + ado MCP
./bootstrap.sh --no-browser    # skip Chrome + chrome-devtools MCP
./bootstrap.sh --check-only    # report dependency status, change nothing

If the toolchain is already present, install the plugins directly. npm install uses npm workspaces and also builds @agentic-loop/core; install.sh then symlinks both plugins and runs the config wizard.

shell
git clone <this-repo>
cd agentic-loop
npm install                    # workspaces + builds core dist/
./install.sh                   # both plugins; or: opencode | claude
Authenticate yourself — installers never do. After install, sign in to the platforms you use: gh auth login for GitHub, az devops login (or an AZURE_DEVOPS_EXT_PAT) for Azure DevOps. The Chrome DevTools MCP launches its own isolated Chrome profile on first use.

Load the plugin

Claude Code

shell
claude --plugin-dir /abs/path/to/claude-plugin

Or add the repo as a marketplace: /plugin marketplace add /abs/path/to/repo then /plugin install agentic-loop.

OpenCode

./install.sh opencode symlinks the agents, commands, skills and a plugin file into ~/.config/opencode (or $OPENCODE_CONFIG_DIR). Re-run after git pull to update — it's idempotent.

Your first run

/agent-loop-task new <idea>→ approve→ loop plans it→ approve-plan→ /agent-loop task <id>→ ship

You author and gate; the loop plans, builds, verifies and reviews in between. On OpenCode, /agent-loop watch polls for approved work; on Claude Code, /agent-loop claim is the one-shot pull.

Setup

The config wizard

./install.sh ends with a short interactive wizard that seeds .agentic-loop.json into the project the loop will drive. Blank always accepts the [default]. It never overwrites an existing file and is skipped under pipes/CI.

--config force the wizard on --no-config skip it -y / --yes write all-defaults {}, no prompts

Every question below is optional — the wizard only writes the keys you set, so a mostly-default config stays tiny.

Q0 Which project directory should the loop drive?

Prompts "Write config for which project directory", defaulting to the current dir. The file is written to <dir>/.agentic-loop.json — the same directory the plugin reads config from at runtime. Skips if it isn't a directory or the file already exists.

writes → <dir>/.agentic-loop.json
Q1 Which code platform do your PRs live on?
[1] GitHub (default)codePlatform: "github"
[2] Azure DevOps (az CLI)codePlatform: "ado"
[3] Azure DevOps (MCP server)codePlatform: "ado-mcp"

Choosing an ADO option then prompts for the required organization URL and project (re-asked until given), an optional repository, and — for ado-mcp only — a required selfLogin for comment filtering.

writes → codePlatform, and ado: {organization, project, repository?, selfLogin?}
Q2 Enable the PR-sitter loop?

Default No. If yes on GitHub, also asks for a "PR search query" (default is:open author:@me). On ADO the sitter watches its own PRs, so no query is asked.

writes → loops.pr-sitter: { enabled: true, query? }
Q3 Run each task in an isolated git worktree?

Default No. If yes, asks for a "Worktrees directory" (default .worktrees) and an optional setup command run in each fresh worktree (e.g. npm ci, since a new worktree has no installed deps).

writes → worktreesDir, worktreeSetup?
Adv Advanced options (single gate)

Default No. Opting in walks three extras:

Task tracker — none / Jira / Azure DevOps. If set, also asks a deep-link base URL (must be http/https) and a default issue type. → projectManagement: {system, baseUrl?, defaultType?}

Review lenses — comma-separated, trimmed, capped at 5. REVIEW then runs once per lens and takes the worst verdict. → reviewLenses: [ … ]

Max iterations — default 3; only written if you give a positive integer other than 3. → maxIterations

The wizard is bash-3.2 compatible and validates the written JSON parses (warns, never fails). Prefer to hand-edit? Everything it writes is documented in the config reference below.

How the loop works

The engineering kind moves a task through four stages. work stages produce edits; check stages record a machine-readable verdict and never edit. Click a stage to inspect it.

01
PLAN
work
02
BUILD
work
03
VERIFY
check
04
REVIEW
check
↩ a FAIL at VERIFY or REVIEW re-fires BUILD — verify and review share one iteration counter, capped at maxIterations (default 3). Cap tripped → the plan is suspect → a human sends it back with /agent-loop-task replan.

The trusted-verdict boundary

A check stage's outcome counts only when recorded through the dedicated loop_verdict tool, and only the stage the loop is currently running may record its own verdict. A stage that writes "PASS" in prose is treated as untrusted narration and ignored. That is what stops a prompt-injected or over-eager stage from waving its own work through.

The pr-sitter kind

TRIAGE check→ FIX work→ VERIFY check→ PUBLISH

Opt-in. Read-only gh/az triage turns each open PR's review comments, failing checks and merge conflicts into findings; FIX addresses them on the PR's existing branch; VERIFY confirms coverage; PUBLISH pushes and replies per finding. It never merges, closes, or approves. PR text is treated as data, not instructions.

Read docs/design/threat-model.md before enabling the PR sitter — it documents the security posture for acting on untrusted PR content.

Gates & the backlog

Authoring and gating live in /agent-loop-task; the loop itself only executes. A task's folder is its status — moving files between folders is how state advances.

The three human gates

Gate 1 · task gateapprove/agent-loop-task approve <id>You accept the goal & acceptance criteria. draft → queued.
Gate 2 · plan gateapprove-plan/agent-loop-task approve-plan <id>You accept the plan the loop parked. plan-review → in-progress. Reject with replan.
Gate 3 · shipship/agent-loop ship <id>You accept the reviewed diff. in-review → completed.

Shortcut: once a task is in the loop's hands, /agent-loop approve advances the single task it's waiting on you for (parked plan → build, or reviewed diff → ship) and /agent-loop reject bounces a parked plan back — no <id> needed unless two+ tasks wait. Draft approval stays the explicit /agent-loop-task approve <id>.

Folder-as-status lifecycle

draft/→ approve → queued/→ PLAN parks → plan-review/→ approve-plan → in-progress/→ REVIEW pass → in-review/→ ship → completed/

Under docs/tasks/ (configurable via tasksDir), plus abandoned/ for dropped work and an ephemeral runs/ holding machine state. PLAN parks — it writes the plan onto the task file and exits rather than blocking on a human, so the loop is always free to pick up other work.

OpenCode vs Claude Code. Same kinds and lifecycle. OpenCode runs a background session.idle driver with a watch polling mode; Claude Code has no background driver — the main agent drives via a bundled MCP server, gates are interactive prompts, and /agent-loop claim replaces watch.

Pausing a run (Esc). On OpenCode, pressing Esc mid-drive interrupts the loop: it stops watch mode and halts after the in-flight stage settles, keeping the run snapshot. Resume exactly where it stopped with /agent-loop recover <id>. That is the difference from /agent-loop stop, a deliberate end that drops the snapshot — nothing to recover.

Config reference

.agentic-loop.json lives at the root of the project the loop drives. Optional — every field has a default. A misconfigured file fails fast with a clear message rather than silently falling back.

FieldDefaultWhat it does
maxIterations3Max iterations before stopping on repeated check-stage failures. Cap tripped → send the task back with replan.
tasksDir"docs/tasks"Root of the task backlog; subfolders are statuses. Also hosts the ephemeral runs/ machine state.
stageTimeoutMinutes60Wall-clock cap on one stage; exceeding it fails the loop instead of hanging.
watchIntervalMinutes5Polling cadence for /agent-loop watch. OpenCode-only — Claude Code has no watch timer.
loops{}Per-kind sections. engineering runs unless disabled; every other kind is opt-in with "enabled": true.
codePlatform"github""github" (gh), "ado" (az CLI), or "ado-mcp" (ADO MCP server). Overridable per kind.
adounset{organization, project, repository?, selfLogin?}. Required for any ADO platform; selfLogin additionally required for ado-mcp.
projectManagementunset{system, baseUrl?, defaultType?}. Pairs local tasks to a Jira / ADO tracker. Pairing is manual — the loop never calls the tracker API.
worktreesDirunsetRun each loop in its own git worktree instead of switching branches. Pair with worktreeSetup.
worktreeSetupunsetShell command run in a fresh worktree, e.g. "npm ci".
reviewLenses[]Run REVIEW once per lens and take the worst verdict. Max 5. Costs ~N× review time.
.agentic-loop.json — example
{
  "codePlatform": "github",
  "worktreesDir": ".worktrees",
  "worktreeSetup": "npm ci",
  "reviewLenses": ["correctness", "security", "test-adequacy"],
  "loops": {
    "engineering": { "enabled": true },
    "pr-sitter": { "enabled": true, "query": "is:open author:@me" }
  }
}

Command cheat-sheet

/agent-loop-task — author & gate

new <idea>Interview you into a draft (splits heavy ideas into sibling drafts).
retask <id> [note]Re-interview & rewrite a draft in place. Draft-only.
approve <id>Task gate → queued.
approve-plan <id>Plan gate → in-progress.
replan <id> [reason]Reject a parked plan → back to queued.

/agent-loop — drive

task <id>Drive one task through its stages.
claimPull & drive the next ready unit of work.
approve [id] · reject [id] [reason]Gate shortcut — advance the one task the loop waits on (parked plan → build, or reviewed diff → ship); reject bounces a parked plan back. Pass id only to disambiguate. (Draft approval stays /agent-loop-task approve.)
watch [interval] · unwatchPoll for work / clear the watch flag (OpenCode only).
statusBacklog + pairing roll-up.
ship <id>Ship gate → completed.
EscPause a running loop — halts after the in-flight stage settles and keeps the snapshot. Resume with recover.
recover <id>Resume an interrupted task (crash or Esc pause) at the exact stage it stopped.
doctor [fix] · stopDiagnose/repair; or halt the run and drop the snapshot (deliberate end — nothing to recover, unlike an Esc pause).
